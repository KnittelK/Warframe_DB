# Warframe Mod Browser

A fully static Warframe mod browser hosted on GitHub Pages. No backend — all
data is pre-merged JSON served alongside the HTML/CSS/JS. Data is automatically
updated nightly from DE's Public Export and warframestat.us.

**Live site:** https://knittelk.github.io/Warframe_DB/

## Features

- Client-side search with multi-term filtering and autocomplete
- Filter presets: Primed, Corrupted, Nightmare, Acolyte
- Quick rarity filters and active filter chips
- Filters: mod type, weapon category, stat type, rarity, drop source
- Detailed mod view with stats at every rank and drop locations
- Grid density toggle (compact / normal / spacious)
- Dark theme inspired by the Warframe aesthetic
- Responsive design for desktop and mobile

## Architecture

```
docs/           Static site (HTML/CSS/JS) deployed to GitHub Pages
  data/
    mods.json   Merged mod data (generated by CI — do not hand-edit)
    meta.json   Build metadata (lastUpdated, modCount)
scripts/
  merge-mods.js Node script that fetches and merges data sources
.github/workflows/
  update-mods.yml   Nightly data fetch + manual trigger
  deploy-pages.yml  Deploys docs/ to GitHub Pages
```

## Data Pipeline

A GitHub Actions workflow runs nightly at 04:00 UTC:

1. Fetches mod data from **DE's Public Export** (`ExportUpgrades.json`) for
   canonical stats
2. Fetches mod data from **warframestat.us** for drop tables, images, and
   descriptions
3. Merges both sources, matching by `uniqueName`
4. Commits the updated `mods.json` to the repository

You can also trigger the data update manually from the Actions tab.

## Local Development

Serve the `docs/` directory with any static file server:

```bash
cd docs && python3 -m http.server 8000
```

To run the data merge locally:

```bash
node scripts/merge-mods.js
```

Requires Node.js 18+ (uses built-in `fetch`).

## Known Issues

### Deploy not re-triggered after data update

`update-mods.yml` pushes with `GITHUB_TOKEN`. GitHub Actions will not fire
another workflow's `on: push` trigger from a `GITHUB_TOKEN` push, so the site
is **not** automatically redeployed with fresh data after each nightly update.

**Fix options:**
- Add a `workflow_run` trigger in `deploy-pages.yml` to run after
  `update-mods.yml` completes.
- Use a repository PAT secret instead of `GITHUB_TOKEN` for the push.
- Merge both workflows: fetch data and deploy in a single workflow.
